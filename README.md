# ShingaDIP

Прототип интеллектуальной системы анализа бухгалтерских операций и первичных документов для автоматизации учета и аудита.

Проект предназначен для демонстрации дипломной работы. Это не промышленная учетная система и не замена бухгалтеру или аудитору. Прототип показывает, как можно совместить:

- загрузку таблицы операций;
- извлечение реквизитов из документов;
- набор формальных контрольных правил;
- поиск аномалий;
- генерацию пояснений с помощью локальной языковой модели.

## Назначение проекта

Система решает учебную задачу: на небольшом наборе операций и документов показать, как искусственный интеллект может помочь в первичном контроле учета.

Прототип умеет:

- принимать бухгалтерские операции из `CSV` и `XLSX`;
- принимать первичные документы в формате `PDF`, `PNG`, `JPG`, `JPEG`;
- извлекать ключевые реквизиты документа;
- сопоставлять запись учета с документом;
- выявлять отклонения и присваивать риск-балл;
- формировать комментарий в стиле помощника аудитора;
- выводить результаты в веб-интерфейсе;
- сохранять отчет локально.

## Как проект работает в целом

Рабочий цикл у проекта такой:

1. Пользователь загружает таблицу операций.
2. Пользователь загружает один или несколько документов.
3. Система нормализует табличные данные:
   - приводит названия колонок к внутреннему формату;
   - приводит суммы к числам;
   - приводит даты к единому виду;
   - проверяет обязательные поля.
4. Система извлекает реквизиты из документов.
5. Для каждой операции подбирается наиболее подходящий документ.
6. Затем выполняются проверки по правилам и ML-анализ.
7. На основе найденных отклонений рассчитывается риск-балл и статус:
   - `OK`
   - `WARNING`
   - `RISK`
8. При включенной локальной LLM формируется текстовое пояснение и рекомендуемое действие.
9. Результаты выводятся в интерфейсе и сохраняются в рабочую директорию.

## Архитектура проекта

Структура проекта:

- `app.py` — интерфейс Streamlit и orchestration всего сценария.
- `shingadip/config.py` — константы, алиасы колонок, веса рисков, служебные пути.
- `shingadip/data_processing.py` — загрузка и нормализация таблицы операций.
- `shingadip/documents.py` — извлечение реквизитов из документов.
- `shingadip/analysis.py` — сопоставление операций с документами, проверки и риск-оценка.
- `shingadip/ai.py` — работа с локальным `LM Studio`, генерация пояснений, итогового заключения.
- `shingadip/reporting.py` — сводка, экспорт отчетов и подготовка данных для отображения.
- `sample_data/` — демонстрационные данные.
- `scripts/generate_demo_documents.py` — генерация простых тестовых PDF-документов.
- `tests/` — базовые unit-тесты.

## Подробно по модулям

### 1. Загрузка и подготовка операций

Модуль: `shingadip/data_processing.py`

Что он делает:

- читает `CSV` и `XLSX`;
- понимает русские и английские названия колонок;
- приводит поля к внутренней структуре;
- создает технические поля:
  - `operation_id`
  - `source_row`
  - `missing_required_fields`
  - отображаемые строки даты и суммы.

Система поддерживает такие основные поля операций:

- дата операции;
- номер документа;
- тип документа;
- контрагент;
- сумма;
- валюта;
- НДС;
- счет учета;
- описание операции;
- ответственный сотрудник.

Обязательные поля для проверки:

- `operation_date`
- `document_number`
- `counterparty`
- `amount`
- `description`

Если обязательное поле отсутствует, это сразу влияет на риск-балл.

### 2. Разбор документов

Модуль: `shingadip/documents.py`

Поддерживаемые типы документов:

- `PDF`
- `PNG`
- `JPG`
- `JPEG`

Система использует несколько путей извлечения данных.

#### Разбор изображений

Если включена галочка `Использовать модель для документов`, изображение отправляется в локальную мультимодальную модель через `LM Studio`.

В текущей конфигурации для этого зафиксирована модель:

- `qwen2.5-vl-7b-instruct`

Модель получает изображение и должна вернуть JSON со следующими реквизитами:

- `document_number`
- `document_date`
- `counterparty`
- `amount`
- `currency`
- `description`
- `extracted_text`

#### Разбор PDF

Для `PDF` сначала извлекается текст через `pypdf`, затем этот текст структурируется через локальную модель документов.

Если модель отработала неполно, система дополнительно пытается вытащить поля регулярными выражениями из уже полученного текста.

#### Резервный путь

Если локальная модель для документов выключена или недоступна, используется fallback:

- для `PDF` — `pypdf + regex`;
- для изображений — `pytesseract + regex`, если в системе установлен `Tesseract OCR`.

Таким образом, проект не зависит полностью от LLM и не падает, если локальный сервер не отвечает.

### 3. Сопоставление операций и документов

Модуль: `shingadip/analysis.py`

Для каждой операции система подбирает наиболее подходящий документ.

Приоритет такой:

1. Совпадение номера документа.
2. Близость суммы.
3. Близость даты.
4. Сходство контрагента.

Если номер документа есть, система в первую очередь ищет точное совпадение номера. Это сделано специально, чтобы не привязывать операцию к чужому документу только из-за похожей суммы или контрагента.

### 4. Проверки по правилам

После сопоставления запускаются формальные проверки.

Система умеет выявлять:

- пустые обязательные поля;
- дубликаты номера документа;
- подозрительное описание операции;
- нетипично большую сумму;
- нетипичного контрагента;
- отсутствие подходящего первичного документа;
- неполные реквизиты документа;
- расхождение суммы операции и документа;
- расхождение даты операции и документа;
- расхождение контрагента операции и документа.

Подозрительные описания ищутся по ключевым словам вроде:

- `срочно`
- `корректировка`
- `вручную`
- `прочее`
- `misc`
- `manual`

### 5. ML-анализ аномалий

Кроме правил используется `IsolationForest` из `scikit-learn`.

Для модели аномалий формируются признаки:

- сумма;
- НДС;
- длина описания;
- длина номера документа;
- число пропущенных обязательных полей;
- частота появления контрагента;
- частота появления счета учета;
- месяц операции;
- день недели.

Если запись выглядит нетипичной относительно остальных, это увеличивает риск-балл.

### 6. Риск-балл и статусы

Каждый тип отклонения имеет собственный вес. Итоговый риск-балл ограничивается диапазоном `0..100`.

Статусы назначаются так:

- `OK` — низкий риск;
- `WARNING` — средний риск или наличие значимых отклонений;
- `RISK` — высокий риск.

Для некоторых важных нарушений введен минимальный порог, чтобы они не оставались в `OK`. Например:

- несоответствие суммы документу;
- отсутствие документа;
- дубликат номера;
- пропуск обязательных полей.

#### Как именно считается риск-балл

В проекте риск-балл считается как сумма баллов за выявленные причины. Затем применяются дополнительные правила калибровки, после чего результат ограничивается диапазоном `0..100`.

Базовые веса причин сейчас такие:

- `missing_required_fields` — `18` баллов
- `duplicate_document_number` — `18` баллов
- `amount_outlier` — `20` баллов
- `suspicious_description` — `10` баллов
- `atypical_counterparty` — `6` баллов
- `no_primary_document` — `8` баллов
- `document_amount_mismatch` — `28` баллов
- `document_date_mismatch` — `14` баллов
- `document_counterparty_mismatch` — `14` баллов
- `document_incomplete` — `6` баллов
- `ml_anomaly` — до `10` баллов по силе аномалии плюс базовая надбавка

Дополнительные правила:

- если отсутствуют обязательные поля, к `18` баллам добавляется еще `2` балла за каждое пропущенное поле, но не более `8` сверху;
- для `ml_anomaly` итоговая прибавка зависит от силы аномалии, рассчитанной моделью `IsolationForest`;
- если документов вообще не загружали, штраф за `no_primary_document` не применяется;
- редкий контрагент сам по себе не считается проблемой и добавляет баллы только если операция уже подозрительна по другим признакам;
- если у операции есть несколько сильных факторов одновременно, применяется дополнительный бонус:
  - `+10`, если найдено 3 сильных фактора;
  - `+16`, если найдено 4 и более сильных факторов.

К сильным факторам относятся:

- отсутствие обязательных полей;
- нетипично большая сумма;
- отсутствие документа при загруженном комплекте документов;
- несоответствие суммы документу;
- несоответствие даты документу;
- несоответствие контрагента документу;
- аномалия, найденная ML-моделью;
- редкий контрагент, если он усиливает уже подозрительную операцию.

После суммирования применяются минимальные пороги для некоторых причин:

- если есть `document_amount_mismatch`, риск-балл поднимается минимум до `45`;
- если есть `no_primary_document`, риск-балл поднимается минимум до `30`;
- если есть одно из нарушений:
  - `document_date_mismatch`
  - `document_counterparty_mismatch`
  - `duplicate_document_number`
  - `missing_required_fields`
  риск-балл поднимается минимум до `30`.

После этого статус назначается по диапазонам:

- `0–29` → `OK`
- `30–69` → `WARNING`
- `70+` → `RISK`

Пример логики:

- если у операции только редкий контрагент, она может остаться `OK`;
- если есть пропуск описания и дубликат номера документа, это уже обычно `WARNING`;
- если одновременно есть крупная сумма, пропуск обязательных полей, редкий контрагент и ML-аномалия, операция, скорее всего, уйдет в `RISK`.

Такая схема сделана специально для снижения ложных срабатываний на больших наборах данных.

### 7. AI-пояснения и итоговое заключение

Модуль: `shingadip/ai.py`

В проекте два независимых AI-режима:

- `Использовать модель для текста`
- `Использовать модель для документов`

Они работают независимо друг от друга.

#### Текстовая модель

Если включена галочка `Использовать модель для текста`, система отправляет в `LM Studio` данные по операциям со статусом выше `OK` и запрашивает:

- комментарий аудитора;
- рекомендуемое действие.

Для текстовых пояснений используется фиксированная модель:

- `qwen2.5-7b-instruct`

Если текстовая модель недоступна, система автоматически использует шаблонные офлайн-комментарии.

#### Итоговое заключение

Когда текстовая модель включена, она также может сформировать краткое итоговое заключение по всему набору операций.

Если модель выключена или недоступна, итог формируется шаблонно.

## Интерфейс приложения

Интерфейс построен на `Streamlit`.

Основные элементы:

- заголовок проекта;
- боковая панель настроек анализа;
- блок загрузки таблицы операций;
- блок загрузки документов;
- кнопка запуска анализа;
- вкладка результатов;
- вкладка детализации операции;
- вкладка документов;
- вкладка итогового отчета.

В сайдбаре можно:

- включить демонстрационный набор;
- подключить демонстрационные документы;
- включить текстовую модель;
- включить модель для документов;
- указать endpoint `LM Studio`;
- ограничить число AI-пояснений;
- ограничить число AI-разборов документов.

## Что сохраняется на диск

При запуске создается рабочая папка в `workspace/`.

Внутри нее создаются:

- копии загруженных документов;
- `reports/audit_results.csv`
- `reports/audit_summary.json`

Это позволяет сохранить результат конкретного запуска отдельно от остальных.

### Файл операций

Поддерживаются форматы:

- `CSV`
- `XLSX`

Рекомендуемые колонки:

- `Дата операции`
- `Номер документа`
- `Тип документа`

Поддерживаются:

- `PDF`
- `PNG`
- `JPG`
- `JPEG`

Документы должны быть читаемыми. Если качество скана плохое, качество извлечения тоже будет ниже.

## Выходные данные

На выходе система формирует:

- таблицу результатов по операциям;
- статус проверки;
- риск-балл;
- причины отклонений;
- комментарий аудитора;
- рекомендуемое действие;
- итоговое заключение;
- экспорт в `CSV` и `JSON`.

## Запуск проекта

### 1. Установка зависимостей

```powershell
pip install -r requirements.txt
```

### 2. Генерация демонстрационных документов

Если папка `sample_data/documents/` пустая, можно создать тестовые PDF так:

```powershell
python scripts/generate_demo_documents.py
```

### 3. Запуск интерфейса

На Windows в этой среде надежнее запускать так:

```powershell
python -m streamlit run app.py
```

Если порт занят:

```powershell
python -m streamlit run app.py --server.port 8502
```

## Подключение LM Studio

Проект использует локальный OpenAI-compatible endpoint.

Базовый адрес по умолчанию:

- `http://127.0.0.1:8080/v1/chat/completions`

### Что нужно сделать в LM Studio

1. Открыть `Developer` -> `Local Server`.
2. Убедиться, что локальный сервер запущен.
3. Загрузить модель `qwen2.5-7b-instruct`, если нужен AI-анализ текста.
4. Загрузить модель `qwen2.5-vl-7b-instruct`, если нужен AI-разбор документов.

### Какие модели используются

Модели в интерфейсе не выбираются вручную. Они зафиксированы в коде:

- текстовая модель: `qwen2.5-7b-instruct`
- модель для документов: `qwen2.5-vl-7b-instruct`

Если нужная модель не загружена в `LM Studio`, приложение покажет предупреждение и использует fallback, если он возможен.

## OCR для изображений

Если режим разбора документов через модель выключен, изображения можно разбирать через `Tesseract OCR`.

Для этого `Tesseract` должен быть установлен локально.

Если его нет:

- приложение не падает;
- просто снижается качество извлечения для изображений;
- анализ операций и отчетность продолжают работать.

## Ограничения проекта

Это учебный прототип, поэтому у него есть ограничения:

- нет интеграции с реальными учетными системами;
- нет обучения модели на данных предприятия;
- правила и веса рисков заданы вручную;
- качество извлечения документов зависит от качества сканов и от локальной модели;
- система не заменяет профессиональное аудиторское суждение.

## Тесты

Запуск базовых тестов:

```powershell
python -m unittest discover -s tests -v
```

Тесты покрывают:

- нормализацию колонок;
- поиск обязательных полей;
- фиксацию несоответствия документа;
- нормализацию ответа локальной модели по документу.
