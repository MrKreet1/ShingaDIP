# ShingaDIP

ShingaDIP — локальный демонстрационный прототип интеллектуальной системы для анализа бухгалтерских операций и первичных документов. Проект показывает, как совместить:

- загрузку учетных данных из `CSV` и `XLSX`;
- извлечение реквизитов из `PDF` и изображений документов;
- контроль по правилам;
- поиск аномалий;
- генерацию пояснений через локальные модели;
- наглядное представление результата в `Streamlit`.

Проект учебный. Он не заменяет бухгалтера, аудитора или промышленную учетную систему.

## Что умеет проект

- загружать операции из `CSV` и `XLSX`;
- загружать первичные документы в формате `PDF`, `PNG`, `JPG`, `JPEG`;
- извлекать реквизиты документа несколькими способами:
  - `PDF text` через `pypdf`;
  - `Tesseract OCR`;
  - `Vision model` через локальный OpenAI-compatible endpoint;
  - `LightOnOCR` как отдельный OCR/backend для документов;
- сопоставлять операции и документы;
- вычислять риск-балл и статус `OK / WARNING / RISK`;
- формировать комментарий аудитора и рекомендуемое действие;
- строить расширенные аналитические отчеты по рискам, причинам, контрагентам и сверке документов;
- сохранять отчет в `CSV` и `JSON`.

## Как работает система

Общий сценарий работы такой:

1. Пользователь загружает таблицу операций.
2. Пользователь загружает один или несколько документов.
3. Система нормализует таблицу:
   - приводит названия колонок к внутренним полям;
   - парсит даты и суммы;
   - отмечает пропущенные обязательные поля.
4. Для каждого документа выбирается способ извлечения текста и реквизитов.
5. Из документа извлекаются:
   - `document_number`
   - `document_date`
   - `counterparty`
   - `amount`
   - `currency`
   - `description`
6. Для каждой операции подбирается наиболее подходящий документ.
7. Запускаются проверки по правилам и ML-анализ аномалий.
8. Рассчитывается риск-балл и присваивается статус.
9. При включенной текстовой модели формируются AI-комментарии.
10. Результаты выводятся в интерфейс и сохраняются в `workspace/`.

## Архитектура

- `app.py` — Streamlit-интерфейс и orchestration всего сценария.
- `shingadip/config.py` — константы, режимы, фиксированные модели, веса рисков.
- `shingadip/data_processing.py` — чтение и нормализация таблицы операций.
- `shingadip/documents.py` — разбор документов, OCR, LightOnOCR, fallback-логика.
- `shingadip/analysis.py` — сопоставление операции с документом, проверки и риск-оценка.
- `shingadip/ai.py` — вызовы локальных OpenAI-compatible endpoint'ов для текста и документов.
- `shingadip/reporting.py` — сводка, экспорт и подготовка данных для отображения.
- `sample_data/` — демонстрационный набор.
- `verification_data/` — отдельные наборы для проверок и нагрузочного сценария.
- `tests/` — unit-тесты базовой логики.

## Подробно по модулям

### 1. Загрузка и подготовка операций

Модуль: `shingadip/data_processing.py`

Поддерживаются русские и английские названия колонок. После чтения система приводит таблицу к внутренней схеме:

- `operation_date`
- `document_number`
- `document_type`
- `counterparty`
- `amount`
- `currency`
- `vat`
- `account`
- `description`
- `responsible_employee`

Дополнительно создаются служебные поля:

- `operation_id`
- `source_row`
- `missing_required_fields`
- отображаемые поля даты и суммы.

Обязательные поля для контроля:

- `operation_date`
- `document_number`
- `counterparty`
- `amount`
- `description`

### 2. Разбор документов

Модуль: `shingadip/documents.py`

Поддерживаемые форматы:

- `PDF`
- `PNG`
- `JPG`
- `JPEG`

Каждый документ на выходе превращается в единый объект со структурой:

```json
{
  "file_name": "...",
  "stored_path": "...",
  "extraction_method": "...",
  "extracted_text": "...",
  "document_number": "...",
  "document_date": "...",
  "counterparty": "...",
  "amount": 0.0,
  "currency": "...",
  "description": "..."
}
```

#### Режимы анализа документов

В интерфейсе доступны режимы:

- `Auto`
- `PDF text`
- `Tesseract`
- `Vision model`
- `LightOnOCR`

Что они значат:

- `Auto`:
  - для `PDF` сначала пытается использовать текстовый слой;
  - если текст слабый или отсутствует, пробует `LightOnOCR`, затем `Vision model`, затем `Tesseract`;
  - для изображений сначала пробует `LightOnOCR`, затем `Vision model`, затем `Tesseract`.
- `PDF text`:
  - использует текстовый слой PDF;
  - если этого недостаточно, переходит к fallback.
- `Tesseract`:
  - использует локальный `Tesseract OCR`;
  - для PDF сначала пытается превратить страницы в изображения.
- `Vision model`:
  - использует фиксированную мультимодальную модель для документов;
  - для PDF с текстовым слоем может структурировать уже извлеченный текст.
- `LightOnOCR`:
  - использует отдельный OCR/backend документов;
  - для `PDF` сначала рендерит страницы в изображения, затем отправляет их в модель;
  - после OCR-текста реквизиты выделяются регулярными выражениями и нормализацией.

#### LightOnOCR

В проекте `LightOnOCR` подключен как отдельный OpenAI-compatible endpoint для документов. Внутри приложения по умолчанию используется идентификатор:

- `lightonocr-2-1b`

Если backend возвращает другой совместимый `model id`, приложение пытается подобрать его автоматически.

Логика такая:

1. Документ превращается в изображение или набор изображений.
2. Страницы PDF рендерятся примерно в `200 DPI`.
3. Длинная сторона изображения ограничивается примерно `1540 px`.
4. Изображение отправляется в `LightOnOCR`.
5. Модель возвращает OCR-текст.
6. Из OCR-текста извлекаются нужные поля.

Такой режим соответствует официальным рекомендациям из model card `lightonai/LightOnOCR-2-1B`: OCR работает по изображению страницы, а для PDF рекомендуется рендеринг порядка `200 DPI` с сохранением геометрии документа.

#### Vision model для документов

Второй AI-режим документов использует фиксированную модель:

- `qwen2.5-vl-7b-instruct`

Она нужна не как основной OCR-движок, а как альтернативный backend:

- для изображений модель может сразу вернуть структурированный JSON;
- для `PDF` с текстовым слоем она может структурировать уже извлеченный текст документа.

#### Fallback-логика

Если выбранный режим не сработал, система не падает. Она:

- логирует предупреждение в поле `warnings`;
- пробует следующий допустимый способ;
- сохраняет фактический способ в `extraction_method`.

Примеры итоговых методов:

- `pdf_text`
- `tesseract`
- `vision_model_text`
- `vision_model_image`
- `lightonocr`
- `fallback`

### 3. Сопоставление операций и документов

Модуль: `shingadip/analysis.py`

Для каждой операции ищется наиболее подходящий документ. Приоритет примерно такой:

1. совпадение номера документа;
2. близость суммы;
3. близость даты;
4. похожесть контрагента.

Если подходящего документа нет, это влияет на риск только тогда, когда документы действительно были загружены. Если пользователь вообще не загрузил комплект документов, система не штрафует все строки массово.

### 4. Проверки по правилам

Система умеет выявлять:

- пропущенные обязательные поля;
- дубликаты номера документа;
- подозрительные описания;
- нетипично большие суммы;
- редких контрагентов как усиливающий фактор;
- отсутствие подходящего документа;
- неполные реквизиты документа;
- расхождение суммы;
- расхождение даты;
- расхождение контрагента.

### 5. ML-анализ аномалий

Кроме правил используется `IsolationForest` из `scikit-learn`.

Модель учитывает:

- сумму;
- НДС;
- длину описания;
- длину номера документа;
- число пропущенных полей;
- частоту контрагента;
- частоту счета;
- месяц операции;
- день недели.

Чувствительность модели ослаблена на больших наборах, чтобы не получать массовые ложные срабатывания.

### 6. Риск-балл и статусы

Итоговый риск-балл ограничен диапазоном `0..100`.

Статусы:

- `0–29` -> `OK`
- `30–69` -> `WARNING`
- `70+` -> `RISK`

#### Как именно считается риск-балл

Базовые веса причин:

- `missing_required_fields` -> `18`
- `duplicate_document_number` -> `18`
- `amount_outlier` -> `20`
- `suspicious_description` -> `10`
- `atypical_counterparty` -> `6`
- `no_primary_document` -> `8`
- `document_amount_mismatch` -> `28`
- `document_date_mismatch` -> `14`
- `document_counterparty_mismatch` -> `14`
- `document_incomplete` -> `6`
- `ml_anomaly` -> `10`

Дополнительная логика:

- за несколько пропущенных обязательных полей начисляются дополнительные баллы;
- `ml_anomaly` учитывает не только сам факт срабатывания, но и силу аномалии;
- редкий контрагент не считается самостоятельной проблемой и усиливает риск только в сочетании с другими сигналами;
- если одновременно найдено несколько сильных факторов, добавляется бонус к риску.

Минимальные пороги:

- `document_amount_mismatch` поднимает риск минимум до `45`;
- `no_primary_document` при загруженных документах поднимает риск минимум до `30`;
- `document_date_mismatch`, `document_counterparty_mismatch`, `duplicate_document_number`, `missing_required_fields` поднимают риск минимум до `30`.

### 7. AI-пояснения и итоговое заключение

Модуль: `shingadip/ai.py`

В проекте два независимых AI-режима:

- `Использовать модель для текста`
- `Использовать модель для документов`

Они не зависят друг от друга.

#### Текстовая модель

Для комментариев аудитора используется фиксированная модель:

- `qwen2.5-7b-instruct`

Она получает данные по операциям со статусом выше `OK` и возвращает:

- краткий комментарий;
- рекомендуемое действие.

Если текстовая модель недоступна, используются шаблонные офлайн-комментарии.

#### Итоговое заключение

При включенной текстовой модели можно получить и сводное текстовое заключение по всему запуску. Если модель недоступна, итог формируется шаблонно.

## Интерфейс приложения

Основные блоки интерфейса:

- заголовок проекта;
- боковая панель настроек;
- загрузка файла операций;
- загрузка документов;
- запуск анализа;
- вкладка результатов;
- вкладка реестра рисков;
- вкладка причин отклонений;
- вкладка контрагентов;
- вкладка сверки документов;
- вкладка детализации операции;
- вкладка документов;
- вкладка итогового отчета.

### Что настраивается в боковой панели

- использование демонстрационного набора;
- подключение демонстрационных документов;
- включение текстовой модели;
- включение модели для документов;
- `Endpoint` текстовой модели;
- режим анализа документов;
- `Endpoint` vision-модели документов;
- `Endpoint LightOnOCR`;
- максимум AI-разборов документов;
- максимум LLM-пояснений.

### Что показывается по документу

Во вкладке документов отображаются:

- фактический способ извлечения;
- уверенность по заполненности реквизитов;
- номер, дата, контрагент, сумма, валюта, описание;
- извлеченный текст;
- предупреждения и информация о fallback.

### Какие дополнительные отчеты доступны

- `Реестр рисков` — операции со статусами `WARNING` и `RISK`;
- `Причины` — количество, доля и средний риск по каждой причине;
- `Контрагенты` — количество операций, общая сумма, риск-профиль и проблемы со сверкой;
- `Сверка документов` — статус `MATCH / MISMATCH / MISSING / NOT_PROVIDED` и совпадение ключевых реквизитов.

## Подключение локальных моделей

### Текстовая модель через LM Studio

Для комментариев и итогового заключения по умолчанию используется endpoint:

- `http://127.0.0.1:8080/v1/chat/completions`

Что нужно сделать:

1. Открыть в `LM Studio` раздел `Developer -> Local Server`.
2. Убедиться, что сервер запущен.
3. Загрузить модель `qwen2.5-7b-instruct`.
4. Включить в интерфейсе флажок `Использовать модель для текста`.

### Vision model для документов

Для альтернативного разбора документов используется фиксированная модель:

- `qwen2.5-vl-7b-instruct`

Она тоже может работать через `LM Studio` на OpenAI-compatible endpoint. В интерфейсе для нее задается отдельный адрес:

- `Endpoint vision-модели документов`

### LightOnOCR

Для OCR документов используется фиксированный идентификатор по умолчанию:

- `lightonocr-2-1b`

Если backend возвращает другой совместимый `model id`, приложение пытается подобрать его автоматически. Для `LM Studio` это важно, потому что API identifier может отличаться от имени модели на Hugging Face.

По умолчанию приложение ожидает `LightOnOCR` на том же локальном сервере:

- `http://127.0.0.1:8080/v1/chat/completions`

Если модель загружена в `LM Studio`, чаще всего достаточно использовать этот же endpoint.

Отдельный вариант запуска — локальный сервер `vLLM`. Пример команды:

```powershell
vllm serve lightonai/LightOnOCR-2-1B --limit-mm-per-prompt "{\"image\": 1}" --mm-processor-cache-gb 0 --no-enable-prefix-caching
```

После этого:

1. Включите флажок `Использовать модель для документов`.
2. Выберите режим `LightOnOCR` или `Auto`.
3. Проверьте `Endpoint LightOnOCR`.

Если `LightOnOCR` недоступен, проект автоматически уйдет в fallback.

## OCR через Tesseract

Если AI-режимы документов выключены или недоступны, проект может использовать `Tesseract OCR`.

Для этого `Tesseract` должен быть установлен локально. Если его нет:

- приложение не падает;
- в карточке документа появится предупреждение;
- система попытается перейти к другим способам разбора или к fallback.

## Ограничения проекта

- это учебный, а не промышленный прототип;
- модели не обучаются на ваших учетных данных;
- качество извлечения зависит от качества скана и доступности локальных моделей;
- веса риска заданы вручную;
- без установленного `pypdfium2` OCR для PDF через изображения недоступен;
- без `Tesseract` недоступен соответствующий OCR fallback.

```powershell
python -m unittest discover -s tests -v
```

- нормализация колонок операций;

- успешный разбор изображения через `LightOnOCR`;
- fallback из `LightOnOCR` в `pdf_text`;
- использование OCR-реквизитов в дальнейшей сверке операции и документа;
- построение реестра риск-операций;
- агрегация причин отклонений;
- построение отчета по контрагентам;
- построение отчета по сверке документов;
- сохранение расширенных CSV-отчетов.
